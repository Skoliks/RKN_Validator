# Project Rules

Проект реализуется как MVP backend-сервиса для первичной технической проверки сайтов на признаки потенциальных правовых рисков.

## Архитектура

Используется слоистая архитектура.

Основные слои:

- API layer
- Services layer
- Analyzers layer
- Infrastructure layer
- Schemas layer
- Core layer

## Главные правила

1. Бизнес-логика не должна находиться в API-роутерах.
2. API-роутеры только принимают запрос, вызывают сервис и возвращают ответ.
3. CheckService является главным координатором проверки.
4. Анализаторы не должны выполнять HTTP-запросы или запускать браузер.
5. Анализаторы работают только с PageData или готовыми результатами инфраструктурного слоя.
6. Анализаторы не должны рассчитывать риск.
7. RiskService не должен анализировать HTML.
8. RiskService работает только с готовыми результатами анализаторов.
9. ReportService не должен выдумывать факты.
10. ReportService формирует отчёт только на основе CheckResult.
11. ReportService не должен писать «сайт нарушает закон».
12. Итоговый CheckResult не должен содержать HTML.
13. HTML может храниться только во внутренней модели PageData.
14. В MVP не используются база данных, фоновые задачи, Telegram-бот, MCP, LLM, авторизация и история проверок.
15. Playwright допускается только как опциональный infrastructure-слой, выключенный по умолчанию. Браузерный слой не должен содержать правовую бизнес-логику, отправлять формы, делать скриншоты или кликать по бизнес-кнопкам.

## MVP

В MVP входят:

- GET /health
- POST /check
- нормализация ссылки
- проверка доступности сайта
- ограниченный обход страниц
- анализ форм
- анализ согласий
- поиск политики обработки персональных данных
- поиск внешних сервисов
- поиск иностранной авторизации
- проверка HTTPS
- определение признаков ориентации сайта на российских пользователей
- расчёт риска по балльной системе
- краткий текстовый отчёт без LLM

## Не входит в MVP

- PostgreSQL
- SQLAlchemy
- Alembic
- Redis
- Celery / Dramatiq
- Playwright как обязательный компонент
- Telegram-бот
- MCP-сервер
- LLM
- frontend
- авторизация пользователей
- история проверок
- массовая проверка сайтов

## Код-ревью

После каждой итерации Codex должен:

1. Запустить тесты.
2. Проверить запуск приложения.
3. Провести код-ревью.
4. Проверить соответствие слоистой архитектуре.
5. Проверить, что код не вышел за границы MVP.
6. Проверить обработку ошибок.
7. Подготовить отчёт о выполнении и код-ревью.
8. Указать изменённые файлы.
9. Указать, можно ли переходить к следующей итерации.
